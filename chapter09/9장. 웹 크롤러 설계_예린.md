# 9장. 웹 크롤러 설계
## 목표
웹 크롤러를 설계하라.
## 웹 크롤러
- 목적: 웹에 새로 올라오거나 갱신된 콘텐츠 탐색
- 활용 예시
  - 검색 엔진 인덱싱: 웹 페이지를 모아 검색 엔진을 위한 로컬 인덱스 생성
  - 웹 아카이빙: 추후 사용 목적으로 장기 보관 위해 정보 수집
  - 웹 마이닝: 인터넷에서 유용한 지식 도출
  - 웹 모니터링: 인터넷 상 저작권/상표권 침해 사례 모니터링
## 1단계. 문제 이해 및 설계 범위 확정
### 기본 동작
  1. 주어진 URL 집합이 가리키는 모든 웹 페이지 다운로드
  2. 다운받은 웹 페이지에서 URL 추출
  3. 추출된 URL들을 다운로드할 URL 목록에 추가하고, i ~ ii 과정 반복
### 요구사항
- 용도: 검색 엔진 인덱싱
- 매달 수집량: 10억 개
- 새로운/갱신된 웹 페이지 고려: O
- 저장 기간: 5년
- 중복 콘텐츠: 무시
### 만족시켜야 할 속성
- 규모 확장성: 병행성 활용
- 안정성: 비정상적 입력이나 환경에 대한 대응
- 예절: 수집 대상에 짧은 시간 동안 너무 많은 요청 X
- 확장성: 새로운 형태의 콘텐츠 지원에 유연
### 개략적 규모 추정
- QPS = 10억 page/30일/24시간/3600초 = 400page/sec
- 최대 QPS = 2 * QPS = 800
- 웹 페이지 평균 크기 = 500k일 때 
  - 10억 페이지 * 500k = 500TB/월
- 5년 보관 위한 저장 용량: 500TB * 12개월 * 5년 = 30PB

#### 시작 URL 집합
- 크롤링 시작점
- 크롤러가 가능한 많은 링크를 탐색할 수 있도록 URL 선정
  - 전체 URL 공간을 작은 부분집합으로 나누는 전략 사용
    - 지역적 특색, 나라별로 분리
    - 주제별로 분리
#### 미수집 URL 저장소
- "다운로드할" URL을 저장 관리하는 컴포넌트
- FIFO 큐
- 크롤링 상태
  - 다운로드할 URL
  - 다운로드된 URL
#### HTML 다운로더
- 인터넷에서 웹 페이지를 다운로드하는 컴포넌트
#### 도메인 이름 변환기
- URL에 대응되는 IP 주소 탐색
#### 콘텐츠 파서
- 파싱과 검증
- 목적: 이상한 웹 페이지 필터링, 저장 공간 낭비 방지
#### 중복 콘텐츠
- 자료구조를 도입하여 데이터 중복을 줄이고 데이터 처리 소요 시간 감소
- 웹 페이지 해시 값 비교
#### 콘텐츠 저장소
- HTML 문서 보관 시스템
- 데이터 유형, 크기, 저장소 접근 빈도, 데이터 유효 기간 고려
- 해당 시스템에서는 디스크와 메모리 동시 사용
  - 데이터 양 많음 -> 디스크 저장
  - 인기 콘텐츠 -> 메모리 적재하여 접근 지연시간 감소
#### URL 추출기
- HTML 파싱하여 링크 추출
- 절대 경로로 변환하여 반환
#### URL 필터
- 크롤링 대상 필터링
  - 특정 콘텐츠 타입이나 파일 확장자 갖는 URL
  - 접속 시 오류 발생하는 URL
  - 접근 제외 목록 포함 URL
#### 이미 방문한 URL
- 이미 방문한 URL과 미수집 URL 저장소에 보관된 URL 추적
- 서버 부하 감소, 무한루프 방지
- 블룸 필터 or 해시 테이블 활용
#### URL 저장소
- 이미 방문한 URL 보관 저장소
### 웹 크롤러 작업 흐름
1. 시작 URL을 미수집 URL 저장소에 저장
2. HTML 다운로더가 미수집 URL 저장소에서 URL 목록 가져옴
3. HTML 다운로더가 도메인 이름 변환기로 IP 주소 찾고, 접속하여 웹 페이지 다운로드
4. 콘텐츠 파서가 다운로드한 HTML 페이지를 파싱하여 올바른 형식의 페이지인지 검증
5. 파싱 및 검증 이후 중복 콘텐츠인지 확인
   1. 이미 저장소에 있는 경우, 처리하지 않고 버림 
   2. 저장소에 없는 경우 저장 후 URL 추출기로 전달
6. URL 추출기가 HTML 페이지에서 링크 추출
7. 추출한 링크를 URL 필터로 전달
8. 필터링된 URL만 중복 URL인지 판별 
   1. 이미 URL 저장소에 보관된 URL이라면 버림 
   2. 저장소에 없는 경우, URL 저장소에 저장하고, 미수집 URL 저장소에도 전달
## 3단계. 상세 설계
- DFS vs BFS
- 미수집 URL 저장소
- HTML 다운로더
- 안정성 확보 전략
- 확장성 확보 전략
- 문제 있는 콘텐츠 감지 및 회피 전략
### DFS vs BFS
- DFS: 그래프 크기가 크면 어느 정도로 깊게 갈지 알기 어려움
- BFS: 큐 -> 한쪽으로 탐색할 URL in, 다른 한쪽으로 out
  - 문제점
    - 한 페이지에서 나오는 링크의 상당수는 같은 서버로 되돌아간다.
      - 같은 호스트의 링크를 병렬로 다운로드 -> 수집 대상 서버 부하 증가
      - 예외 없는 크롤러
    - 우선순위 없음
      - 모든 웹 페이지가 같은 수준의 품질, 중요성을 가지지 않음
      - 페이지 순위, 사용자 트래픽 양, 업데이트 빈도 등 여러 척도 비교하여 우선순위 구별 필요
### 미수집 URL 저장소
- 예의있는 크롤러, 우선순위와 신선도를 구별하는 크롤러 구현
#### a. 예의
- 짧은 시간 안에 많은 요청을 보내지 않는다.
- 동일 웹 사이트에 대해서는 한 번에 한 페이지만 요청한다.
- 웹 사이트의 호스트명과 다운로드를 수행하는 작업 스레드 사이의 관계 유지
- 각 다운로드 스레드는 별도 FIFO 큐를 가지고 해당 큐에서 꺼낸 URL만 다운로드

<img width="500" alt="image" src="https://github.com/user-attachments/assets/45d81544-6069-4759-9116-8f03b3be2115">

- 큐 라우터: 같은 호스트에 속한 URL은 언제나 같은 큐로 가도록 보장
- 매핑 테이블: 호스트명과 큐 사이 관계 보관
- FIFO 큐: 같은 소트에 속한 URL은 언제나 같은 큐에 보관
- 큐 선텍기: 큐를 순회하며 지정된 작업 스레드에 전달
- 작업 스레드: 순차적으로, 일정 지연시간을 두고 URL 다운로드 작업 수행
#### b. 우선순위
- 척도: 페이지랭크, 트래픽 양, 갱신 빈도 등

<img width="500" alt="image" src="https://github.com/user-attachments/assets/987aee9a-9429-4e30-830f-a0d11eb0c043">

- 순위결정장치: URL 우선순위 정하는 컴포넌트
- 큐 선택기: 순위가 높은 큐에서 더 자주 꺼내도록 구현

#### c. 신선도
- 주기적 재수집
- 최적화 전략
  - 웹 페이지 변경 이력 활용
  - 우선순위 활용

#### d. 저장장치 유형
- 메모리와 디스크 사이 절충안
- 대부분 디스크 적재하되 IO 비용 줄이기 위해 메모리 버처에 큐 배치
- 버퍼에 있는 데이터는 주기적으로 디스크에 기록

### HTML 다운로더
#### Robots.txt
- 웹 사이트가 크롤러와 소통하는 표준적 방법
- 크롤러가 수집해도 되는 페이지 목록 포함
- 웹 사이트 긁기 전에 해당 파일에 나열된 규칙 먼저 확인
#### 성능 최적화
1. 분산 크롤링: URL 공간은 작은 단위로 분할하여 각 서버는 그 중 일부의 다운로드 담당하도록 설계
2. 도메인 이름 변환 결과 캐시: DNS 요청의 동기적 처리로 인한 성능 병목을 캐싱으로 개선
3. 지역성: 크롤렁 작업 서버를 지역별로 분산하여 다운로드 시간 감소
4. 짧은 타임아웃: 대기 시간 길어지는 것을 방지
### 안정성 확보 전략
1. 안정 해시: 다운로더 서버들에 부하 분산 시 적용 가능한 기술
2. 크롤링 상태 및 수집 데이터 저장: 장애 복구 목적
3. 예외 처리: 예외에도 시스템이 중단되지 않도록 처리
4. 데이터 검증: 시스템 오류 방지 목적 
### 확장성 확보 전략
- 새로운 형태의 콘텐츠를 쉽게 지원할 수 있도록 구현
- 모듈을 분리하여 확장성 확보

### 문제 있는 컨텐츠 감지 및 회피
1. 중복 콘텐츠: 해시나 체크섬을 사용하여 중복 콘텐츠 탐지
2. 거미 덫: 무한루프 유도하는 웹 페이지
   - URL 최대 길이 제한
   - 수작업으로 덫 찾아서 크롤러 탐색 대상에서 제외
3. 데이터 노이즈: 의미 없는 콘텐츠 제외

## 4단계. 마무리
### 추가 고려사항
- 서버 측 렌더링: 동적 생성 링크 추적을 위해 페이지 파싱 전에 서버 측 렌더링을 적용
- 원치 않는 페이지 필터링: 스팸 방지 컴포넌트 추가
- 데이터베이스 다중화 및 샤딩: 데이터 계층 가용성, 규모 확장성, 안정성 향상
- 수평적 규모 확장성: 수평적 규모 확장성 달성 위해 무상태 서버로 구현
- 데이터 분석 솔루션
